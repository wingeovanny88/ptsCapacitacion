01. Teoría resumida y mapa conceptual

Modelo mental para entender Frame Banking como plataforma de transacciones.

Lectura ejecutiva

1. Qué debe quedar claro desde el inicio

Frame Banking se explica mejor como una ruta transaccional completa. La transacción entra por un canal, pasa por un servicio digital, cruza reglas de negocio, ejecuta lógica y termina en un core o servicio externo con evidencia observable.

Entrada

Canal, contexto y necesidad de negocio.

Decisión

DFS, operación, reglas, límites y flujos.

Salida

Respuesta, posting, trazas y soporte RUN.

2. Caso de uso

Objetivo de la prelectura

Alinear lenguaje antes de entrar al workshop

Este módulo no busca agotar la teoría. Busca que todos lleguen a la sesión entendiendo cómo mirar una transacción y cómo ubicar las piezas principales de la plataforma.

Pregunta guía

¿Qué recorre el dinero? No solo un endpoint.

¿Qué decide el sistema? Reglas, límites y flujos.

¿Qué necesita soporte? IDs, estado y trazabilidad.

3. Flujo conceptual de una transacción

4. Reglas de negocio

Marco 01

Dominios visibles

La plataforma se organiza por capacidades como cuentas, acuerdos, canales, servicios digitales, límites, transferencias, pagos y posting.

Marco 02

Piezas que deben conectarse

Un servicio digital no vive solo: se relaciona con operaciones, canales, cargos, acuerdos, atributos y lógica de negocio.

Marco 03

Secuencia para explicar

Caso, flujo, reglas, integraciones y RUN. Ese orden mantiene la conversación clara y evita perderse en detalles prematuros.

Entrada

Capa media

Orquestación

Reglas

Decisión

Core

Salida

Mambu o APIs externas ejecutan, responden y dejan evidencia.

6. Ejecución RUN

Checklist mínimo para una transacción +
  1. Identificar canal y business channel.
  2. Identificar digital service y operation.
  3. Validar acuerdo, permisos y estado del servicio.
  4. Validar límites/acumuladores si aplica.
  5. Ejecutar operación o flujo.
  6. Buscar logs por msgId, referencia externa, transaction id o equivalente disponible.
  7. Validar estado, posting y respuesta externa.

7. Monitoreo

Qué mirar después de ejecutar

Estado: éxito, rechazo, timeout o compensación.

Contexto: canal, DFS, operación y regla aplicada.

Evidencia: IDs, posting, respuesta externa y tiempos.

8. Práctica guiada: explicar una transacción en 7 minutos

Dinámica: cada grupo toma una operación real o simulada y la explica sin abrir código durante los primeros 7 minutos.

  1. Identificar canal, DFS, operación y endpoint.
  2. Explicar qué reglas aplican: límites, cargos, acuerdos o posting.
  3. Marcar dónde entra Designer y dónde entra una API externa.
  4. Definir qué IDs usarían para buscar la transacción en logs.
  5. Presentar una hipótesis de falla y cómo la comprobarían.
