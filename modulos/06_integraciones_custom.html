06. Integraciones externas y flujos custom

Patrones seguros para integrar APIs externas desde Designer y Frame Banking.

Vista guiada del módulo

1. Integrar bien es convertir una dependencia externa en un flujo entendible

Integración externa y flujos customizados: custom-service, digital-service, extension points, UTILS, APIs externas, autenticación, business channel, logs y RUN. Para developers, la clave es ver cómo un contrato externo se traduce en una experiencia de negocio legible.

Contrato

Define el acuerdo entre el flujo interno y el sistema externo.

Mapeo

Convierte request y response en estructuras útiles para el dominio.

Soporte

Los errores deben dejar señales claras para RUN.

2. Caso de uso

Qué se construye

Flujos que invocan APIs externas o componentes del Designer

Estos flujos validan, enriquecen, autorizan, consultan, ejecutan o notifican una operación. Lo importante es separar la intención funcional del detalle técnico de la integración.

Lectura práctica

Necesidad: por qué hace falta integrar.

Contrato: qué se envía y qué se recibe.

RUN: cómo se soporta la ruta en producción.

4. Reglas de negocio

Regla 01

Antes de integrar

Confirmar si existe API oficial o custom service, y definir contrato, autenticación, timeout y fallback.

Regla 02

Durante el diseño

Mapear request interno a request externo, normalizar errores y propagar correlation id.

Regla 03

Después de integrar

El payload sensible debe tratarse con cuidado y el flow debe quedar fácil de observar y diagnosticar.

Antes de integrar

  • Confirmar si existe API oficial o custom service.
  • Definir contrato: método, headers, body, response y errores.
  • Definir autenticación sin exponer secretos.
  • Separar validación funcional de falla técnica.
  • Definir timeout, retry y fallback con el equipo correspondiente.

Durante el diseño

  • Mapear request interno → request externo.
  • Mapear response externo → estructura interna.
  • Normalizar errores.
  • Propagar correlation id.
  • Registrar payload sensible solo de forma segura o enmascarada.

6. Ejecución RUN

Runbook para API externa +
  1. ¿El request llegó al flow?
  2. ¿El mapeo construyó el payload esperado?
  3. ¿La autenticación/token fue válido?
  4. ¿La API externa respondió o hubo timeout?
  5. ¿El código externo fue traducido correctamente?
  6. ¿La respuesta permite continuar, rechazar, enviar a MFA o compensar?
  7. ¿El log tiene correlación suficiente para soporte?

7. Monitoreo

Señales que conviene vigilar

Latencia: si la integración tarda, el error operativo cambia.

Timeout / retry: deben estar definidos antes de producir.

Corrrelation y cambios: ayudan a sostener el flujo cuando el contrato evoluciona.

8. Práctica guiada: diseñar integración con API externa

Reto: construir el contrato de integración antes de implementarlo.

  1. Definir endpoint, método, headers, body y respuesta esperada.
  2. Definir autenticación sin exponer secretos.
  3. Mapear respuesta externa a respuesta interna.
  4. Separar error funcional, error técnico y timeout.
  5. Definir qué pasa si la API externa responde tarde después de una operación monetaria.

Patrón documentado: integración Mambu

El publisher DU_INTEGRACION_MAMBU documenta un patrón de integración con Mambu compuesto por:

  • MAMBU_CONNECTOR: ejecuta REST Web Service con métodos POST, GET, DELETE y PATCH.
  • MAMBU_FL_POST_DEBIT_DISPATCH y MAMBU_FL_POST_CREDIT_DISPATCH: despachan hacia MAMBU_CONNECTOR.
  • MAMBU_FL_POST_DEBIT_PARSE_RESPONSE y MAMBU_FL_POST_CREDIT_PARSE_RESPONSE: parsean respuesta.
  • MAMBU_FL_POST_DEBIT_CREATE_MSG y MAMBU_FL_POST_CREDIT_CREATE_MSG: preparan mensajes de débito/crédito.

Seguridad: se detecta parámetro APIKEY. En capacitación se debe nombrar como secreto y nunca copiar su valor.
