08. API Enablement con Postman

Cómo invocar, probar y diagnosticar APIs de servicios financieros digitales usando el caso DEPOSITO_CNB_DEUNA.

Vista guiada del módulo

1. Probar por Postman sirve para validar configuración, no solo conectividad

API Enablement para servicios financieros digitales: cómo exponer, invocar y diagnosticar una operación desde Postman cuando se crea o reutiliza un servicio financiero digital en Frame Banking / DeUna.

El ejemplo trabajado corresponde a transfer-internal + digital-service + posting + flujo custom Mambu para DEPOSITO_CNB_DEUNA.

Antes

Confirmar DFS, operación, canal, flow y posting.

Durante

Enviar un request consistente y controlar IDs de trazabilidad.

Después

Leer respuesta, confirmaciones, postings y logs relacionados.

2. Caso de uso

Qué debe aprender el developer

Invocar una API también es leer el sistema detrás

Postman ayuda a comprobar que la operación está realmente publicada, que el servicio enruta bien y que la respuesta deja señales suficientes para soporte y monitoreo.

Chequeo mínimo

Config: servicio, operación, canal y flow.

Contrato: headers, path y payload coherentes.

RUN: IDs, estado, confirmaciones y posting.

Seguridad: nunca documentar ni subir tokens Bearer reales. En los ejemplos el token se reemplaza por <BEARER_TOKEN>.

3. Flujo: de DFS a Postman

Identificar DFS Confirmar servicio y operación objetivo.
Habilitar canal Verificar permisos y relación con el request.
Preparar contrato Path, headers y payload deben ser coherentes.
Ejecutar Enviar y capturar IDs, estado y respuesta.
Leer RUN Interpretar posting, confirmaciones y trazas.

Lectura funcional del endpoint

POST /RSAdaptorFE/api/v2/own-channels/debit-transfer/internal/online/single-currency/third-account/account/{ResourceID}/transfer
Parte Lectura funcional Qué valida el developer
own-channels Invocación por canal propio. Que el canal esté permitido.
debit-transfer/internal Transferencia interna con débito. Que la operación corresponda al servicio configurado.
online/single-currency Operación online y una sola moneda. Que no se esté configurando un caso diferido o multimoneda.
third-account/account/{ResourceID} Cuenta/recurso origen. Que el ResourceID apunte a la cuenta esperada.
/transfer Acción transaccional. Que existan permisos, canal, límites, flow y posting.

4. Reglas de negocio

Lectura 01

Un endpoint no demuestra por sí solo que el caso está listo

Antes de probar, conviene confirmar que el servicio existe, que la operación está activa y que el canal realmente puede invocarla.

Lectura 02

El request debe contar una historia coherente

Headers, path, body e IDs de negocio deben apuntar al mismo escenario funcional y técnico.

Lectura 03

La respuesta debe poder leerse en RUN

Si el request no deja trazabilidad suficiente, la prueba queda incompleta aunque responda exitosamente.

Antes de probar

  1. El digital service existe y está activo.
  2. La operación está habilitada.
  3. El business channel está asociado; en el ejemplo: APP_CNBS_IN.
  4. Los headers coinciden con el contrato esperado.
  5. El payload contiene headerRQ, securityRQ y messageRQ.
  6. messageRQ.digitalService coincide con la configuración: DEPOSITO_CNB_DEUNA.
  7. Existen parámetros custom/posting/flow asociados cuando el servicio los requiere.

Qué no conviene asumir

  • Crear el DFS no implica publicar automáticamente cualquier endpoint.
  • El canal del header debe estar habilitado para la operación.
  • El nombre del servicio en el body no enruta si no existe configuración detrás.
  • No todos los reversos/postings son iguales.
  • No exponer tokens, JWT o credenciales.

Header/bloque Uso Riesgo típico
Content-Type Media type PTS esperado por el adaptador. Parsing inválido o rechazo técnico.
pCHANNEL Canal de negocio de entrada. No coincide con securityRQ.channelId o no está habilitado.
Authorization Bearer token/JWT. Token expirado, scope insuficiente o exposición accidental.
headerRQ.msgId Correlación del request. Reutilizar IDs dificulta trazabilidad.
messageRQ.digitalService Servicio financiero digital a ejecutar. Servicio inexistente, inactivo o sin flow/posting.
general.transactionId Referencia/idempotencia de negocio. Duplicidad, reintentos inseguros o conciliación difícil.
beneficiaries[].account.othersId Cuenta destino en Mambu. Cuenta inexistente, bloqueada o tipo incorrecto.

6. Ejecución (RUN)

Qué debería poder explicar el participante

Estado: si la respuesta fue exitosa y por qué.

Movimiento: cómo se ven débito, crédito y confirmaciones.

Soporte: qué IDs usar para seguir la operación en logs.

Cómo leer la respuesta exitosa

Campo Qué indica Uso operativo
statusRS.code = 0 Respuesta general exitosa. Primer filtro de salud.
messageRS.responses[0].isError = false Respuesta funcional sin error. Confirmar antes de revisar posting.
PTSId Identificador PTS. Buscar en trazas/logs.
roundId Ronda de procesamiento/posting. Agrupa confirmaciones y pasos.
transactionId ID transaccional generado/propagado. Correlación end-to-end.
confirmations[] Lista de postings/confirmaciones. Validar débito y crédito.
postingId ID de cada confirmación de posting. Diagnóstico por ordinal/línea.
operation Operación concreta ejecutada. Validar ruta correcta.
Ordinal 10

Débito

La primera confirmación del ejemplo corresponde a WITHDRAWAL, amount = -3 y operación ...INTERNAL_TRANSFER.DEBIT.

Ordinal 20

Crédito

La segunda confirmación corresponde a DEPOSIT, amount = 3 y operación ...INTERNAL_TRANSFER.CREDIT.

Checklist Postman

  1. Generar msgId nuevo por request.
  2. Generar/usar transactionId controlado.
  3. Validar que pCHANNEL coincida con securityRQ.channelId.
  4. Confirmar que messageRQ.digitalService existe y está activo.
  5. Validar cuenta origen en path y cuenta destino en body.
  6. Ejecutar con monto bajo en ambiente dev.
  7. Validar statusRS.code, isError, PTSId, roundId, transactionId.
  8. Validar confirmaciones por ordinal.
  9. Buscar logs por msgId, transactionId, PTSId, paymentOrderId y postingId.

7. Monitoreo

Correlación

headerRQ.msgId, headerRS.msgIdOrg, transactionId, PTSId, paymentOrderId.

Posting

roundId, postingId, ordinal, operation, confirmationNumber.

Core/Mambu

transactionChannelId, ID_MAMBU_DEBIT, RESMAMBU, saldos, bookingDate y valueDate.

Para DEPOSITO_CNB_DEUNA, monitorear CONTEXT.CUSTOM.TRANSACTION_ID, paymentOrderId, transaction_id, transaction_pts_id y MSG_OUT.

8. Práctica guiada: Postman como herramienta de diagnóstico

Reto: ejecutar o simular el curl sanitizado y explicar la respuesta sin depender de memoria.

  1. Renovar msgId y transactionId.
  2. Confirmar coherencia entre pCHANNEL y securityRQ.channelId.
  3. Identificar cuenta origen en path y cuenta destino en body.
  4. Validar statusRS.code, isError, PTSId, roundId y transactionId.
  5. Explicar el débito ordinal 10 y el crédito ordinal 20.
  6. Documentar qué buscar en logs si falta una confirmación.
