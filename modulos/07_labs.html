07. Laboratorios prácticos

Ejercicios propuestos para capacitación, onboarding y práctica guiada.

Vista guiada del módulo

1. Los laboratorios convierten la teoría en práctica controlada

Laboratorios prácticos sobre digital-service, limits, accumulators, Designer, custom-service, merchant-payment, cash-in, cash-out, posting y RUN. El objetivo es que el participante pueda moverse desde la idea hasta la evidencia.

Explorar

Entender cómo se combina cada componente.

Practicar

Resolver escenarios guiados con criterios claros.

Demostrar

Entregar diagrama, request, trazas y diagnóstico.

2. Caso de uso

Cómo usar esta sección

Pasar de teoría a práctica sin perder contexto

Estos ejercicios están diseñados para workshops de 2 a 4 horas o como ruta de onboarding técnico. Cada laboratorio debe dejar una señal clara de comprensión y una señal clara de soporte.

Lo que se espera

Entender: qué problema resuelve el flujo.

Construir: qué piezas intervienen.

Explicar: qué evidencia deja en RUN.

3. Flujo de laboratorio

Lab 1Leer una transacción end-to-end.
Lab 2Configurar mentalmente un DFS.
Lab 3Diagnosticar un rechazo por límite.
Lab 4Diseñar un flow en Designer.
Lab 5Integrar una API externa ficticia.
Lab 1

Leer una transacción end-to-end

Objetivo: identificar canal, DFS, operación, reglas, flow, integración y logs.

  1. Elegir un caso: CNB retiro o Merchant Payment.
  2. Dibujar el flujo en papel.
  3. Marcar puntos de validación.
  4. Definir qué buscar en RUN.
Lab 2

Configurar mentalmente un DFS

Objetivo: que el dev pueda explicar qué componentes deben existir.

  1. Definir operación.
  2. Seleccionar business channel.
  3. Asociar agreement/cuentas.
  4. Definir cargos, límites y flow.
Lab 3

Diagnóstico de rechazo por límite

Objetivo: investigar un rechazo sin culpar al endpoint.

  1. Identificar operation y DFS.
  2. Consultar acumulador.
  3. Consultar límite y consumo.
  4. Determinar si corresponde rechazo o SFA/MFA.
Lab 4

Diseñar flow en Designer

Objetivo: diseñar un flujo custom observable.

  1. Entrada.
  2. Validación.
  3. Llamada externa.
  4. Decisión.
  5. Salida y logs.
Lab 5

Integrar API externa ficticia

Objetivo: practicar mapeo y errores.

  1. Crear contrato mock.
  2. Mapear request/response.
  3. Definir timeout y error funcional.
  4. Agregar observabilidad.
Lab 6

Runbook de incidente

Objetivo: armar respuesta operativa ante error productivo.

  1. Restate del problema.
  2. Dominio y flujo.
  3. Hipótesis documentadas.
  4. Checks RUN.
  5. Monitoreo y próximos pasos seguros.

4. Reglas de negocio

Regla 01

Sin contrato confirmado no hay atajo

Cuando se requiera un dato exacto de endpoint o payload, conviene revisarlo en la documentación oficial o en el publisher correspondiente.

Regla 02

El lab debe cerrar en evidencia

No basta con entender la idea; hay que mostrar diagrama, request, respuesta y lectura operativa.

Regla 03

RUN es parte del ejercicio

Si el equipo no puede explicar cómo lo monitorea, el laboratorio todavía no está completo.

Estos labs deben apoyarse en contratos confirmados. Cuando se requiera un dato exacto de endpoint o payload, conviene revisarlo en la documentación oficial de APIs o en el publisher correspondiente.

5. Integraciones

DFS

Contexto

Ubica el caso de uso y las reglas asociadas.

Designer

Flujo

Ordena pasos, decisiones y errores.

API externa

Integración

Permite practicar mapeo, timeout y fallbacks.

RUN

Soporte

Obliga a pensar en trazabilidad y monitoreo.

Los labs integran DFS, Designer, límites/acumuladores, APIs externas, UTILS, Mambu/APIM/PFTR como conceptos. El ambiente real debe validar nombres exactos, URLs, credenciales, permisos y estados.

6. Ejecución RUN

Formato de entrega por equipo:
1. Diagrama del flujo.
2. Componentes Frame Banking usados.
3. Configuración requerida.
4. Reglas de negocio identificadas.
5. APIs/servicios externos.
6. Cómo probar.
7. Cómo monitorear.
8. Qué riesgos quedan pendientes de validar.

7. Monitoreo

Señales de observabilidad

Status: éxito, rechazo, timeout o compensación.

Latencia: cuánto tarda cada punto del flujo.

Trazas: IDs, servicio externo, posting y reconciliación.

Cada laboratorio debe terminar con un dashboard o checklist de observabilidad: status, error, latencia, trace, servicio externo, límite/acumulador, posting y reconciliación si aplica.

8. Cierre práctico: evaluación por evidencia

El participante aprueba si puede demostrar, con evidencia, cómo una operación viaja de Postman a Frame Banking y cómo se monitorea.

  1. Presenta un diagrama del flujo.
  2. Presenta el curl sanitizado.
  3. Explica configuración requerida.
  4. Marca IDs de trazabilidad.
  5. Propone dos fallas y cómo las diagnosticaría.

Lab adicional: reverse engineering de un publisher

Objetivo: que el equipo lea un `.publisher` y construya una explicación operativa sin inventar comportamiento.

  1. Elegir uno: DU_INTERNAL_TRANSFER, DU_INTEGRACION_MAMBU o DU_LIMITS_CUSTOM_FLUJOS.
  2. Identificar flows, REST operations, DFS, posting, acumuladores y parámetros.
  3. Marcar secretos o datos sensibles que no deben copiarse.
  4. Dibujar el flujo técnico.
  5. Preparar una prueba Postman o pseudoprueba.
  6. Definir checks RUN.
  7. Listar lo que queda pendiente de validar.
