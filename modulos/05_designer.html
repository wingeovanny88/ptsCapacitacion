05. Designer

Cómo crear diagramas y conectarlos a servicios financieros digitales.

Vista guiada del módulo

1. Designer convierte reglas en pasos visibles

Designer se relaciona con digital-service, custom-service, business logics, dynamic flows, extension points, integraciones externas y RUN. La lectura útil no empieza por los nodos, sino por la operación que el flow quiere resolver.

Diseño

Un flow debe resolver un caso concreto.

Observabilidad

Cada paso debe poder rastrearse en logs y trazas.

Soporte

Los labels y la estructura deben ayudar a diagnosticar.

2. Caso de uso

Qué se construye

Un diagrama de flujo que implemente una operación real

La práctica no se centra en decorar nodos. Se centra en validar, transformar, llamar servicios internos o externos, decidir errores y dejar trazabilidad suficiente para RUN.

Regla de lectura

Un nodo: una responsabilidad.

Un flow: una historia clara.

Un error: una salida entendible.

3. Flujo para crear un diagrama útil

1. Nombrar el caso

Ejemplo: validar cuenta origen antes de pago comercio.

2. Definir contrato de entrada/salida

Qué datos recibe, qué datos necesita, qué devuelve y qué errores controla.

3. Modelar pasos atómicos

Cada nodo debe hacer una cosa: validar, transformar, llamar API, decidir, persistir o responder.

4. Conectar con DFS

El flow debe quedar invocable desde el servicio/operación o punto de extensión configurado.

5. Preparar RUN

Labels, logs, correlation id, errores legibles y puntos de reintento.

4. Reglas de negocio

Diseño 01

Buenas prácticas

Un nodo, una responsabilidad. Los labels deben ayudar a soporte, no solo al desarrollo.

Diseño 02

Errores a evitar

No ocultar llamadas externas, no hardcodear rutas y no reintentar operaciones monetarias sin revisar estado.

Diseño 03

Señal de calidad

Si el flow se puede explicar en 30 segundos, normalmente también se puede soportar mejor.

Buenas prácticas de diseño

  • Un nodo, una responsabilidad.
  • Nombrar labels pensando en soporte, no solo desarrollo.
  • Controlar ramas de error.
  • No ocultar llamadas externas dentro de expresiones difíciles de auditar.
  • Evitar secretos en configuraciones visibles.
  • Propagar identificadores de correlación.

Errores comunes

  • Crear flujos sin RUN plan.
  • No distinguir error funcional vs error técnico.
  • No mapear respuestas externas.
  • Hardcodear canales, códigos o rutas sin catálogo/parametrización.
  • Reintentar operaciones monetarias sin consultar estado.

7. Monitoreo

Qué observar en el flow

Inicio y fin: ayudan a medir alcance y duración.

Decisión y error: muestran dónde se corta la ruta.

Llamada externa: permite ubicar fallas de integración o latencia.

Todo flow debe tener puntos observables: inicio, fin, llamada externa, decisión de negocio, error funcional, error técnico y salida final. Los labels en Designer deben ayudar a soporte a ubicar el problema.

8. Práctica guiada: construir un flow observable

Reto: dibujar un flujo Designer para validar una cuenta beneficiaria antes de ejecutar una operación monetaria.

  1. Entrada: request normalizado.
  2. Validación: campos obligatorios y cuenta beneficiaria.
  3. Integración: llamada a servicio externo o UTILS.
  4. Decisión: continuar, rechazar o responder error técnico.
  5. Observabilidad: labels, correlation id, logs de entrada/salida y error.

MANAGEMENT_INT_TRANSFER

Flow publicado como REST operation. Ejecuta validaciones de cuenta, consulta límites por operación, consulta acumuladores y decide entre servicio sin MFA o con MFA según segmento y límites documentados.

MAMBU_FL_CHANNEL_TRANSFERENCIA_INTERNA

Arma estructura de mensaje para canal de transferencia interna y estructuras para reversa manual/ordenada y automática. También contiene actividad de business log.

MAMBU_CONNECTOR

Flow con llamadas REST a Mambu usando métodos POST, GET, DELETE y PATCH. Usa API key como header; no debe exponerse.
